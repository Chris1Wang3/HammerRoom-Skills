# 需求评审裁决方法手册（Review Playbook）

## A. 方法总览（目标 / 四问 / 不做的事）

ReviewCourt 输出的是基于会议证据的「评审建议」，不是代替产品、技术负责人或评审委员会作出正式决策。

只回答四个问题：

1. 会议是否出现明确评审结论？
2. 结论由哪些原话直接支撑？
3. 仍有哪些条件阻止需求进入开发或验收？
4. 若提供 PRD，文档是否准确承接会议结论？

避免四种失败：

1. 把智能纪要当成逐段转写原话
2. 没有正式结论却输出通过 / 有条件通过 / 驳回
3. 臆造 Owner、DDL、数字、范围或验收口径
4. 扩展为通用会议总结、人员评价或项目管理系统

---

## B. 证据协议（标签 · 索引 · 充分性）

### B1. 证据标签

| 标签 | 定义 | 可支撑内容 |
|------|------|------------|
| `[原话]` | 逐段转写中可定位的发言 | 裁决、质疑、答复、承诺、Owner、DDL、验收口径 |
| `[文档]` | PRD 中可定位的明确文本 | 已写需求、规则、范围、指标、验收标准 |
| `[推断]` | Agent 根据多条事实作出的归纳 | 风险归类、建议优先级、验收标准草案 |
| `[缺口]` | 所需事实不存在或不可验证 | 待确认项、补证动作 |

智能纪要只能作为「导航线索」，不得标成 `[原话]`。若逐段转写与智能纪要冲突，以逐段转写为准，并在报告中指出冲突。

### B2. 证据索引

为每条使用到的证据分配稳定编号：

```text
E-001 [原话]
说话人：<姓名或会议中显示名称>
位置：<HH:MM:SS 或 paragraph_id>
原话：<最短但语义完整的原文>
用途：<裁决 / 质疑 / 答复 / 阻塞 / Owner / DDL / 验收>
```

PRD 证据使用相同编号体系：

```text
E-012 [文档]
位置：<章节 / 标题 / 条目 / 页码>
原文：<最短但语义完整的文档内容>
用途：<范围 / 规则 / 指标 / 验收>
```

### B3. 证据充分性

一条证据可以支撑多个相关判断，但不得跨越其原意。例如：「这个方案先别上」可以支撑暂停或驳回，不能支撑永久取消。

裁决证据优先级：

```text
明确决定原话
> 带条件的放行原话
> 决策者对结论的复述确认
> 智能纪要摘要
> Agent 推断
```

只有前三类可以支持「通过 / 有条件通过 / 驳回」。

## C. 评审事实抽取（事实卡 · 五类问题）

按议题而不是按发言顺序整理信息。每个议题建立一张最小事实卡：

| 字段 | 规则 |
|------|------|
| 议题 | 使用中性、具体的短语 |
| 质疑 | 谁对什么提出什么问题 |
| 答复 | 谁作答、答复是否覆盖质疑 |
| 状态 | 已关闭 / 部分关闭 / 未关闭 / 无法判断 |
| 影响 | 是否阻止开发、联调、发布或验收 |
| 证据 | 引用 `E-xxx` |

重点抽取以下五类问题：

1. **业务价值**：目标用户、问题强度、指标和收益是否明确。
2. **需求范围**：MVP、非目标、边界条件和异常路径是否明确。
3. **技术可行性**：依赖、数据、性能、兼容性、降级与发布方案是否明确。
4. **体验与运营**：关键流程、文案、配置、客服和运营承接是否明确。
5. **测试与合规**：验收口径、数据安全、权限、审计和法规要求是否明确。

不要因为某类问题未被讨论就主动替会议补充答案；把它列为风险或 `[缺口]`。

## D. 四态裁决规则（通过 / 有条件通过 / 驳回 / 信息不足）

按下表自上而下核对。裁决状态只能有一个。

| 状态 | 必要条件 | 禁止条件 |
|------|----------|----------|
| 通过 | 存在明确放行原话；不存在仍开放的 P0 阻塞项 | 仅有「看起来可以」「大家没意见」等模糊表达 |
| 有条件通过 | 存在「满足条件即可进入下一阶段」的明确原话；条件可识别 | 只有未解决问题，但没有任何放行意图 |
| 驳回 | 存在明确否决、暂停、退回重做或不进入下一阶段的原话 | 仅有强烈质疑但无正式结论 |
| 信息不足 | 无明确裁决，或转写/权限/上下文不足 | 不得为了给答案而选择其他状态 |

### D1. 通过

可接受表达示意：

- 「这个版本通过，可以进入开发。」
- 「没有阻塞项，按当前方案推进。」

仍需检查会议后半段是否出现撤销、改口或新增 P0 阻塞。

### D2. 有条件通过

可接受表达示意：

- 「这两项补齐后就可以开工。」
- 「原则通过，前提是灰度和降级方案在开发前确认。」

必须列出每个条件的关闭标准。Owner 或 DDL 未说出时标「待确认」，不能因此把条件删除。

### D3. 驳回

可接受表达示意：

- 「本次不通过，重新设计后再评审。」
- 「先暂停，业务价值没有验证清楚。」

区分「本版本驳回」「暂缓」和「需求永久取消」，不得扩大原话范围。

### D4. 信息不足

以下任一情况直接判信息不足：

- 只有讨论、建议或投票倾向，没有正式结论；
- 只有智能纪要，没有可核验逐段转写；
- 关键结论被截断，无法确认上下文；
- 多名参会人意见相反，决策角色未收口；
- 会议不是需求、PRD 或技术方案评审。

录制或转写无权限时：先按 **J. 异常与降级** 走申请流程（prepare → 用户确认 → commit），审批完成前停止裁决；仅当用户放弃申请、审批失败或仍无可用转写时，再判信息不足。

## E. 场景 A：会议单源裁决

按以下顺序执行：

1. 写出本次被评审的需求和版本范围。
2. 建立证据索引。
3. 将质疑与答复配对，标记关闭状态。
4. 识别决定原话及说话人角色。
5. 按四态规则输出唯一裁决。
6. 整理阻塞条件、待确认项和验收标准草案。

若同一场会议评审多个需求，分别裁决；不要用一个总体结论覆盖所有议题。最终可增加会议级汇总：全部通过 / 部分可推进 / 整体不可推进 / 信息不足。

---

## F. 场景 B：会议与 PRD 双源核对

先独立完成会议裁决，再读取 PRD。不要先看 PRD 后反向解释会议原话。

### F1. 对照单位

以最小可执行需求项为单位核对，例如：

- 一个业务规则；
- 一个页面或交互状态；
- 一个接口、数据或性能要求；
- 一个异常、降级或兼容策略；
- 一个验收标准。

### F2. 差异状态

| 状态 | 定义 | 建议动作 |
|------|------|----------|
| 已对齐 | 会议结论与 PRD 语义一致 | 保留，必要时补证据引用 |
| PRD 遗漏 | 会议已明确，PRD 找不到对应内容 | 补入指定章节 |
| 内容冲突 | 会议与 PRD 对同一事项给出不同口径 | 以会议最新明确结论为候选，要求 Owner 确认并修订 |
| PRD 新增 | PRD 出现会议未讨论的新范围或规则 | 标记「未过会」，决定是否补评审 |
| 双方不明 | 会议和 PRD 都不足以确认 | 列为待确认，不猜测 |

### F3. 差异矩阵

每项差异必须同时引用会议和 PRD 位置；某侧确实没有内容时写 `[缺口]`。

| 需求项 | 会议结论 | PRD 当前内容 | 状态 | 影响 | 建议修改 | 证据 |
|--------|----------|--------------|------|------|----------|------|
| <最小需求项> | <结论或缺口> | <内容或缺口> | <五态之一> | <开发/体验/测试/合规> | <章节级动作> | E-xxx, E-yyy |

不要直接改写用户的 PRD，除非用户在完成审查后另行明确授权。

---

## G. 阻塞项与验收标准

### G1. 阻塞等级

| 等级 | 定义 |
|------|------|
| P0 | 未关闭前不得进入开发、联调、发布或验收 |
| P1 | 不阻止当前阶段，但必须在指定里程碑前关闭 |
| P2 | 优化建议或后续版本事项 |

若会议没有明确阻塞等级，Agent 可以按影响提出 `[推断]` 等级，但必须保留原始表述并允许用户调整。

### G2. 阻塞项字段

每项至少包含：

```text
问题：<具体未决事项>
等级：<P0/P1/P2；推断时标注>
关闭标准：<如何证明已解决>
Owner：<原话明确的人；否则待确认>
DDL：<原话明确的时间；否则待确认>
证据：<E-xxx>
```

### G3. 验收标准

只把会议或 PRD 已明确的口径写成正式验收标准。Agent 为提高可执行性补写的内容，标题必须标「验收标准草案 `[推断]`」。

使用 Given / When / Then：

```text
Given <前置条件>
When <用户行为或系统事件>
Then <可观察、可验证的结果>
```

避免「体验良好」「性能正常」「符合预期」等不可验证表达。缺少阈值、环境或范围时明确列为待确认。

## H. 输出格式锁（快速版 / 审计版模板）

### H1. 分区规则

- 快速版分区数量、顺序、标题保持与下方模板一致，禁止新增 / 删减 / 重排
- 场景 A 不输出空的「PRD 差异」章节
- 结论必须使用「评审建议」，不得写成 AI 代表团队的正式决定
- 每条关键裁决、阻塞项、差异项必须挂 `E-xxx`

### H2. 快速版

```markdown
# 需求评审裁决单

## 评审建议

**状态：** 通过 / 有条件通过 / 驳回 / 信息不足  
**理由：** <一句话，不超过 50 字>  
**直接证据：** E-xxx

## 进入下一阶段前的条件

| 等级 | 条件 | Owner | DDL | 关闭标准 | 证据 |
|------|------|-------|-----|----------|------|

## 关键质疑与答复

| 议题 | 质疑 | 答复 | 状态 | 证据 |
|------|------|------|------|------|

## PRD 差异

<仅场景 B 输出差异矩阵；场景 A 不显示空章节>

## 验收标准草案

1. Given ... When ... Then ... `[原话/文档/推断]`

## 待确认项

- <谁需要确认什么；未知身份时写「待指定决策人」>

## 证据索引

- E-001 `[原话]` <说话人> · <时间/段落>：「<原文>」
- E-002 `[文档]` <章节>：「<原文>」
```

### H3. 审计版

审计版在快速版基础上增加：

1. 会议与材料范围；
2. 裁决规则逐项核对；
3. 全量议题事实卡；
4. 完整 PRD 差异矩阵；
5. 风险与补证计划；
6. 证据充分性说明。

不要为了「详细」重复粘贴整段转写。只保留能证明判断的最小完整原话。

---

## I. 文风护栏

- 去 AI 化：禁止「综上所述」「值得注意的是」「全面提升」「深度赋能」「形成闭环」等空话
- 每段只表达一个核心判断，优先「结论句 + 证据句 + 动作句」
- 禁止连续使用「首先 / 其次 / 最后」流水句式
- 禁用表达：「看起来没问题」「大家没意见就算过」「优化一下再说」
- 避免堆砌形容词；一句一个信息点，优先短句与列表
- 单段以 3–5 行为上限；多议题对比优先用表
- Owner、DDL、数字、范围未明确时写「待确认」，禁止脑补

---

## J. 异常与降级

| 异常 | 处理 |
|------|------|
| 多场候选会议 | 展示主题、时间、会议号，等待用户选择 |
| 多个录制文件 | 根据时间和内容范围筛选；无法判断时让用户选择 |
| 无录制 | 判信息不足，建议获取逐字稿或会议记录 |
| 无权限 | prepare → 展示申请信息 → 用户确认后 commit；审批完成前停止裁决，不直接判信息不足 |
| 审批中 | 停止裁决，提供审批链接和后续重试建议；审批失败或用户放弃后再判信息不足 |
| 只有智能纪要 | 可整理议题，不输出通过 / 有条件通过 / 驳回 |
| 转写缺段 | 标注缺失范围；若影响结论则判信息不足 |
| 发言人无法识别 | 保留时间 / 段落位置，姓名写「未识别发言人」 |
| PRD 无法读取 | 完成会议单源裁决，跳过差异核对并说明原因 |
| PRD 有多版本 | 要求用户指定评审对应版本，不自行选择 |
| 会议中无正式结论 | 判信息不足，列出需要决策人确认的问题 |

## K. 交付前质量闸门（含输出前自检）

### K1. 完整性检查

- 是否形成「原话 → 裁决 → 条件 / 动作」闭环
- 非「信息不足」裁决是否至少有一条直接支持状态的 `[原话]`
- Owner / DDL / 数字 / 范围是否零臆造
- 场景 B 的每项差异是否同时标出会议证据与 PRD 位置或缺口
- 模型补写内容是否全部标为 `[推断]` 或「草案」

### K2. 反越权检查

命中任一项须重写或停止：

1. 把智能纪要冒充为逐段转写原话
2. 输出未包含证据编号的关键裁决
3. 自动修改 PRD、外发消息、创建任务或员工评价
4. 输出 Token、内部 `meeting_id` 或无关个人信息

### K3. 生成前一致性自检（强制）

输出 Markdown 前逐项核对，不通过先修正：

1. 会议确实已经结束且属于需求、PRD 或技术方案评审
2. 多候选会议或多版本 PRD 已由用户确认
3. 裁决状态唯一且使用「评审建议」表述
4. 非「信息不足」裁决至少有一条直接支持状态的 `[原话]`
5. 智能纪要没有被冒充为逐段转写原话
6. 每项 P0 阻塞都有关闭标准和证据
7. Owner、DDL、数字和范围没有臆造
8. 场景 B 的每项差异都同时标出会议证据与 PRD 位置或缺口
9. 模型补写内容全部标为 `[推断]` 或「草案」
10. 输出未包含 Token、内部 `meeting_id` 或无关个人信息
11. 未执行自动修改 PRD、外发消息、创建任务或员工评价
12. 格式一致：快速版分区齐全且顺序不变（场景 A 省略空的 PRD 差异章）

---

## L. 规则优先级

1. **证据与裁决规则**（最高）：原话优先、四态唯一、不补事实
2. **输出格式锁**：本文件 H 章节
3. **方法规则**：本文件 A–G、J 章节
4. **文风约束**：本文件 I / K 章节

冲突时：保留高优先级规则，调整低优先级的表达方式。
