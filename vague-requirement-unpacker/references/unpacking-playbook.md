# 模糊需求接招方法手册（Unpacking Playbook）

## Layer 0 模糊度判定

判断三个问题：
1. 当前材料模糊到什么程度：一句方向、半结构化口头需求、会议纪要短句、还是已接近 PRD。
2. 紧急度：今天回话 / 本周方案 / 下 sprint。
3. 交付档位：L1 / L2 / L3。

输出格式：
```text
判定：模糊方向 / 半结构化需求 / 已接近 PRD
紧急度：...
建议档位：...
保守假设：...
不适用提醒：...
```

## Layer 1 意图还原

从原话拆出：
- 表层任务：字面上要做什么
- 可能真实意图：要结果、要方案、要验证、要对齐、要背锅材料
- 不想要什么：过度设计、拖延、无法落地、跨部门扯皮等
- 交差标准：今天能回什么、会议上能对齐什么、文档里要有什么

输出必须包含两段话术：
- 30 秒回话版：直接、短、可发送
- 稍正式版：适合会议或文档开头

## Layer 2 场景收敛

只保留最多 3 个核心场景。每个场景用：
```text
角色：谁
触发：在什么情况下
目标：想完成什么
验收：做到什么算过
```

如果超过 3 个场景，先分期，不要把所有场景塞进一期。

## Layer 3 边界骨架

输出模块清单、权限边界和 MoSCoW。

模块颗粒度建议：
- 用户侧能力
- 运营/管理侧能力
- 数据/报表能力
- 配置/规则能力
- 风控/合规能力
- 通知/触达能力

MoSCoW 必须包含：
- Must：不做就无法对齐对方目标
- Should：本期应该有，但可降级
- Could：有价值但可后置
- Won't：本期明确不做

## Layer 4 流程状态

主路径优先，避免堆异常。

输出：
```text
主流程：A -> B -> C -> D
关键状态：状态名 / 触发条件 / 下一步 / 责任方
卡点：最可能堵在哪里
```

如适合用 Mermaid，可输出简短流程图；否则用文本流程。

## Layer 5 风险补丁

L3 默认打开，L1/L2 按需打开。

包含：
- 异常分支：失败、撤销、超时、重复、权限不足、合规拦截
- 埋点指标：漏斗、转化、使用、效率、风险
- 非功能：性能、稳定性、安全、隐私、审计、可配置性
- 依赖：数据源、权限、接口、运营流程、法务合规

## Handoff 到评审模拟器

当用户说“继续评审预演”“帮我过会前压测”“看看研发/运营/老板会怎么挑战”时，把 L2/L3 输出作为输入，转 `pm-requirement-review-simulator`。

交接摘要应包含：
```text
需求名
业务目标
核心场景
能力边界
不做范围
关键约束
待确认项
当前风险红旗
```
